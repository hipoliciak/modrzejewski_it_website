You’ve done the heavy lifting. You’ve enabled the required editions—Enterprise, Unlimited, or those supported by Data 360, utilizing the new features within Marketing Cloud Growth or Advanced Edition. You’ve configured the trigger to listen to updates on a Prospect, Lead, or Contact. Your flow logic is flawless.

But then you click Debug in Flow Builder, and the first thing you see in the execution log is a crushing sight:

`$Record = null`

If you are experiencing this, you aren't doing anything wrong—you’ve simply run into a known platform limitation. This post will explain how this powerful new feature works, why standard debugging fails, and the precise steps you need to take to successfully test and validate your flow.

## A New Era of Personalized Automation

Automation Event-Triggered Flows represent a significant step forward in uniting the power of the core Salesforce Platform with real-time marketing events.

When a Prospect, Lead, Contact, or a related individual record is created or updated within Data 360 or Marketing Cloud Growth/Advanced, it fires an asynchronous event. This flow allows you to capture that event instantly and react with the full toolkit of Flow Builder—from decision trees to email actions. It's the engine for real-time personalization.

## The Problem: Why Standard Debug Fails

The traditional Flow Debugger operates by allowing you to manually select a record or inject a sample JSON payload. However, Automation Event-Triggered Flows are unique. They are triggered by external, asynchronous event payloads managed by Data 360 or the Marketing Cloud Data Bus.

When you click the "Debug" button from the parent flow canvas, the debugger attempts to start the interview but lacks the mechanism to simulate or "mock" that complex incoming Data 360 payload. Consequently, the `$Record` global variable (which should hold the triggering Prospect, Lead, or Contact data) is instantiated as `null`.

Salesforce formally acknowledges this behavior in Knowledge Article `#005185008`. Per the documentation, debugging these specific Automation Events evaluates each step as an isolated transaction that rolls back automatically, preventing standard, integrated debugging from the parent canvas.

## The Solution, Part 1: Debugging the Logic (The Subflow Hack)

To test your automation logic, you must decouple it from the event trigger. If you debug the parent flow and try to pass `$Record` to a subflow, you are still passing `null`.

Here is the correct approach to test your execution steps:

### Step 1: Create an Autolaunched Flow

Take all the logic you intended to have in your parent flow (the "Send Email" action, any Update Records elements, decisions, etc.) and move it into a new, standard Autolaunched Flow (No Trigger).

### Step 2: Define a Record Input Variable

In this new Autolaunched flow, create a record variable designed to hold your primary record (e.g., `var_IncomingProspect` for a Prospect object). Crucially, ensure that **Available for input** is checked.

`[INSERT_IMAGE_CREATE_RECORD_VARIABLE_AVAILABLE_FOR_INPUT]`

### Step 3: Reference the Subflow

Go back to your Automation Event-Triggered Flow (the one with `$Record` as null). Replace all the logic with a single element: a Subflow that calls your new Autolaunched flow. In the input mapping configuration, map the parent’s empty `$Record` to your `var_IncomingProspect` input variable.

### Step 4: Debug the Autolaunched Flow DIRECTLY

This is the key step. You must completely ignore the parent flow’s debug button.

Open your Order Shipped Subflow (the Autolaunched flow) in its own Flow Builder window.

Click **Debug** from inside this flow.

The configuration screen will open, but now it will prompt you to manually select a specific test record to populate `var_IncomingProspect`.

Select a sample record and click **Run**. The flow will execute perfectly, allowing you to validate your outcomes (like email sends) without needing the live Data 360 event.

## The Solution, Part 2: Live End-to-End Testing

The Subflow method confirms your logic works if it receives a record. But how do you confirm that the real-time event from Data 360 is actually firing the flow in production?

Because these events run asynchronously in the background under system users, watching the Flow Builder canvas won't show you the live execution. You must track the system execution directly.

### Step 1: Activate Both Flows

Activate your Autolaunched (Subflow) and your parent Automation Event-Triggered Flow. End-to-end integration testing can only happen when the flows are live.

### Step 2: Set a Trace Flag for "Automated Process"

This is the most critical monitoring step. You cannot trace your own user account; you must trace the background system user handling the event bus.

Navigate to **Setup > Debug Logs**.

Click **New** to create a Trace Flag.

Set the **Traced Entity Type** to **Automated Process** (in some organizations, this might also be your designated **Default Workflow User**).

Configure a Debug Level where the **Workflow** category is set to at least `INFO` (use `FINER` if you need to inspect the raw JSON event payload coming from Marketing Cloud/Data 360).

`[INSERT_IMAGE_DEBUG_LOGS_SETUP_AUTOMATED_PROCESS]`

### Step 3: Trigger the Event and Check Logs

Go to your source system (Data 360 or Marketing Cloud Growth) and perform the action that triggers the automation event (e.g., updating a Contact's email address).

Return to the **Setup > Debug Logs** page in Salesforce and refresh. You will see a new debug log generated by the **Automated Process** user. This log contains the definitive, step-by-step transcript showing the flow’s execution, confirming the event reached Salesforce, the flow ran, and whether the subflow succeeded or failed.

### Step 4: Verify Outcomes

Check the destination record’s Activity Timeline. Since your flow includes a "Send Email" action, you should see a successful email entry logged against the Contact or Lead, marked as created by the **Automated Process** system user.
